# X への投稿

## 2つの動作モード
- **明示テキストあり**（ユーザーが本文を指定）：機密ガードだけ確認し、そのまま投稿する。
- **自動モード**（スケジュール起動など、本文指定なし）：下の手順で自分で考えて投稿要否を判断する。

## 実行環境の注意
- `rg` は利用できません。探索には `find` / `grep` / `tail` / `sed` などの標準的なコマンドを使う。必要な確認は、この SKILL.md に書かれた具体的なコマンドを優先する。

## 自動モードの手順
1. まず本日の投稿数を確認する：
   exec で `grep -c "^$(date +%F)" {baseDir}/posted.log 2>/dev/null || echo 0`
   結果が 12 以上なら、何もせず「今回は投稿なし」と返して終了。
2. 投稿前チェック: message ツールの read アクションで slack から対象チャンネル(REDACTED)の直近メッセージを取得し、現在の話題を把握する。
   - 入力例: {"action": "read", "channel": "slack", "channelId": "REDACTED", "limit": 15 }
   - 件数（limit）指定で新しい順に返る。なるべく取得後に各メッセージの ts（Unix秒）が「現在時刻 − 6時間」以降のものだけを文脈として使う形で絞る。
   - 「現在時刻 − 6時間」以降のものがない場合は取得できたメッセージから文脈を絞る
   - slack ツールが使えない、または取得が空で直近文脈が得られない場合は、古い／推測の文脈で投稿せず見送る。
3. そのテーマを「公開情報に対する一般論」へ変換した投稿候補を考える。
4. 下の編集方針と機密ガードでセルフチェックする。
5. 過去の投稿履歴を確認する：
   exec で `tail -n 30 {baseDir}/posted.log 2>/dev/null || echo none`
   これまで投稿したテーマ・主張を把握しておく。
6. 安全で価値ある候補があれば投稿する。新しい話題がない、または候補が過去投稿と重複する場合は、まず下の「引用ポストモード」を検討し、それも成立しなければ「スレッド展開モード」を検討する。どちらも成立しなければ「今回は投稿なし」と返す。

## 引用ポストモード（タイムラインの良い投稿に視点を添える）
自動モードで Slack に新しい話題がないとき、スレッド展開モードより先にこのモードを検討する。ホームタイムラインの投稿から「自分の視点を1つ足せるもの」を選び、引用ポストする。他のアカウントとの自然な接点を作るためのモードであり、足せる視点がなければ無理に引用しない。

1. 本日の引用数を確認する：
   exec で `grep -c "^$(date +%F).*quote=" {baseDir}/posted.log 2>/dev/null || echo 0`
   結果が 2 以上なら、このモードは行わずスレッド展開モードへ進む。
2. タイムラインの取得：
   exec で `python3 {baseDir}/scripts/fetch_timeline.py --limit 30`
   1行1件の JSON（id, author, name, text, created_at, likes, reposts）が新しい順に返る。自分の投稿・リツイート・返信・引用の入れ子は除外済み。
3. 引用候補の選定：次の条件をすべて満たす投稿だけを候補にする。
   - 技術・ビジネス・開発現場の話題で、自分の視点・経験的な一般論・実践的な応用を1つ足せる
   - センシティブな話題（政治・宗教・社会的論争・炎上中の議論・個人への攻撃）に関わらない
   - **同一アカウント7日ルール**：exec で `grep "quote_author=<author>" {baseDir}/posted.log | tail -n 1` を確認し、直近7日以内に同じアカウントを引用していたら選ばない
4. 引用文の基準：
   - **賛同・補足・展開のみ**。引用元への批判・反論・添削・採点は、内容が正しくても書かない。
   - 引用元の本文のコピー・言い換え・要約はしない（引用カードとして本文は表示されるため不要）。自分の視点だけを書く。
   - 「持ち帰れる気づき／明日試せるtips」の基準は通常投稿と同じ。書けないなら引用しない。
   - 編集方針・機密ガード・他者への配慮は通常投稿と同一に適用する。
5. 投稿の実行：
   python3 {baseDir}/scripts/post.py --quote <ツイートID> --quote-author <投稿者のauthor値> "<引用文>"
6. 引用は1日最大2件で、1回の実行では1件まで。日次上限（12件）にも合算する。良い候補がなければ引用せずスレッド展開モードへ進む（見送りは正常な結果）。

## スレッド展開モード（新しい話題がないときの代替）
自動モードで「新しい話題がない」「候補が既存投稿と重複する」と判断したときに限り、過去の自分の投稿にスレッド（自己リプライ）で返信し、テーマを深掘りしてよい。1回の実行で返信できるのは**最大2件**、それぞれ**別の元投稿**に1件ずつとする。品質基準を満たす候補が1つしかなければ1件、なければ0件でよい。**見送りは正常な結果**であり、件数を埋めるために基準を緩めないこと。

1. 候補の選定：exec で次を実行し、直近7日以内の元投稿（返信ではない行）からランダムに5件を候補として取得する：
   `awk -F'\t' -v d="$(date -d '7 days ago' +%F)" '$1 >= d && $4 == ""' {baseDir}/posted.log | shuf | head -n 5`
   投稿日はログの1列目、ツイートIDは2列目の URL（`https://x.com/i/status/<ID>`）末尾の数字。この5件の中から「まだ語り残しがあるテーマ」を最大2つ選ぶ（必ず別々の元投稿であること）。よい候補がなければ見送る。
2. スレッドの現状確認：exec で `grep "reply_to=" {baseDir}/posted.log 2>/dev/null || echo none` を実行し、ログの4列目 `reply_to=<ID>` が候補の元投稿（またはその返信）を指す行を数える。
   - **元投稿1つにつき自分の返信は最大5件まで**。すでに5件あるテーマは選ばない。
   - **4件目以降の返信は新規性のハードルを一段上げる**。スレッドの後半はネタが薄くなりやすいので、これまでの返信と明確に違う角度を出せるときだけ続ける。
   - 返信先はスレッドの**最後尾**のツイートID。元投稿への自分の返信がすでにあるなら、その中で最新のもののIDに返信する（最後尾以外に返信するとスレッドが枝分かれする）。
3. 内容の基準：元投稿の言い換え・要約・補足程度のものはNG。「新しい具体例」「反対側の考慮点」「実践時の落とし穴」「一歩進めた応用」など、元投稿を読んだ人が**追加で持ち帰れるもの**を最低1つ含める。作れないならスレッド展開もしない。
4. 編集方針・重複の回避・機密ガード・他者への配慮のルールは、通常投稿とまったく同じように適用する。
5. 2件投稿するときは、1件目の投稿後に exec で `sleep 60` を実行してから2件目を投稿する（連続投稿の間隔を空ける）。
6. スレッド返信も1日の投稿上限（12件）にカウントする。手順1の日次チェックはこのモードでも省略しない。本日の残り枠が1件なら、返信も1件までにする。

## 編集方針（ここが今回の肝）
- **非属人化**：誰が発言したか・社内の誰の意見かは一切出さない。「社内で」「チームで」といった出所も書かない。
- **公開情報への一般化**：社内の話題はあくまで着想のきっかけとして扱い、投稿は「すでに公知の事柄に対する自分の一般的な考察・知見」として成立させる。Slackを読んでいない第三者が読んでも自然に意味が通ること。
- **記事参照モード（スレッドが特定の公開記事に言及しているとき）**：対象スレッドが特定の公開記事に言及していて、その記事URLが特定できる場合に限り、そのURLを投稿に含めてよい。
  - 含めてよいのは公開記事のURLだけ。本当に公開コンテンツを指していると確認できないURL、社内リンク、認証が要るページ、短縮URL、出所不明のURLは含めない。確認できないときはURLを付けず通常モードで扱うか、見送る。
  - 投稿の主役は「その記事を起点にした自分の気づき・着眼点」。記事の要約で終わらせず、「この切り口がおもしろい」「ここに気づくと見え方が変わる」と思わせる視点を1つ立て、その面白さを前面に出す。
  - スレッドの議論は着眼点の"きっかけ"としてだけ使う。議論の引用・言い換え・要約、誰が何を言ったか、社内の固有事情は一切出さない（非属人化・公開情報への一般化の原則はそのまま適用）。
  - スレッドを見ていない第三者が、リンク先の記事と投稿文だけで意味が通り、かつ「読んでみたくなる」状態にする。
  - 1投稿につき記事URLは1つまで。記事と気づきの関連が薄い、または十分におもしろい着眼点が立てられないときは、URLを付けず（または投稿せず）見送る。「中身のある一言を」の基準（持ち帰れる気づき／明日試せるtips）はこのモードでも満たすこと。
- **価値の提供（発見・気づき・応用Tips）**：Slackの話題を取っ掛かりに、そこから得た発見や気づき、またはビジネスやシステム開発の現場で応用できそうな実践的なtipsや考え方を提案する。読者が「これは使える」と感じる具体性を持たせること。ただし、その具体性は一般論・公知の知識の範囲で組み立て、社内の事情・数値・固有の事例には依存させない。
- **一個人の所感のトーン**：会社の公式見解と誤解されない書き方にする。
- **中身のある一言を**：抽象的な感想や当たり障りのない一般論で終わらせず、1投稿につき「持ち帰れる気づき」か「明日試せるtips」を最低1つは含める。それが作れないときは投稿しない。

## 重複の回避
- 過去に投稿した内容と、テーマ・主張・切り口が実質的に重複するものは投稿しない。
- 言い回しや語尾を変えただけの焼き直しも「重複」とみなして避ける。
- 同じ話題を再び扱うのは、明確に新しい視点・情報・角度を加えられるときだけ。
- 十分に新しいと言い切れないなら、投稿しない方を選ぶ。

## 絶対に投稿しない（機密ガード）
- 社内でしか知り得ない情報、未発表の製品・機能・スケジュール・数値・顧客/取引先
- Slackの文面の引用や言い換え（要約でもNG。テーマだけを抽出する）
- 個人を特定できる情報や、特定の人物の発言の再現
- 公開記事以外のURL（社内ツールのリンク、認証が必要なページ、短縮URL、出所不明・未確認のURL）。※ 「記事参照モード」で扱う、公開記事のURLのみ例外として許可する。

## 他者への配慮（必須）
- 特定の個人・団体への誹謗中傷、名誉を傷つける内容は投稿しない。
- 人種・民族・国籍・性別・性的指向・宗教・障害・年齢などに基づく差別的・侮辱的な表現は一切しない。
- 他者を貶める、嘲笑する、見下す、攻撃する内容は投稿しない。
- 競合他社や特定の人物を名指しで批判・揶揄しない。
- 読み手を不快にさせる、煽動的・攻撃的・挑発的なトーンを避ける。皮肉や決めつけにも注意する。
- センシティブな政治・宗教的論争で一方を断じるような物言いはしない。
- 少しでも誰かを傷つける・不快にさせる可能性があれば、投稿しない方を選ぶ。

## 投稿の実行
投稿すると決めたら exec で次を実行する（本文は必ず1つの引数として渡す）：
python3 {baseDir}/scripts/post.py "<投稿本文>"
スレッド展開モードのときは返信先ツイートIDを指定する：
python3 {baseDir}/scripts/post.py --reply-to <返信先ツイートID> "<投稿本文>"
出力が `OK <URL>` なら成功。`post failed` や `missing env vars` が出たらその内容を報告する。

## 頻度
- 1日の投稿は最大12件（スレッド返信を含む）。価値ある一般化ができないときは投稿しない方を選ぶ。
- 投稿件数はノルマではない。実行のたびに投稿する必要はなく、「今回は投稿なし」は失敗ではなく正常な判断のひとつ。
