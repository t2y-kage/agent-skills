# X の読み取り（ダイジェスト作成と返信の提案）

## このスキルの役割
このスキルには 2 つのユースケースがある。どちらも**読み取り専用**で、投稿・返信・引用・いいね・リポストは行わない（書き込みは x-post スキルの領分）。結果は実行結果としてそのまま返す（Slack などの外部サービスへは送らない）。

1. **ダイジェスト作成**（既定）：タイムライン（ホーム／最近検索）を取得し、技術・ビジネスの観点で価値のある話題を選んで要約ダイジェストを作る。
2. **返信の提案**：指定されたアカウント（1 つでも複数でもよい）のタイムライン、または指定されたポストを読み、返信する価値のあるポストを選んで返信の下書きを提案する。**提案するだけで、このスキルからは返信を投稿しない**。

## 実行環境の注意
- `rg` は利用できません。探索には `find` / `grep` / `tail` / `sed` などの標準的なコマンドを使う。必要な確認は、この SKILL.md に書かれた具体的なコマンドを優先する。

## 取得モードの使い分け
fetch_timeline.py にはタイムライン系の 3 モード（home / search / user）と、単一ポスト取得の post モードがある。目的が違うので、実行のたびにどれを使うかを選ぶ。

|  | home（ホームタイムライン） | search（最近検索） | user（特定アカウント） |
|---|---|---|---|
| 何が取れるか | フォロー中アカウントの最新投稿（逆時系列） | 直近 7 日の公開ポスト全体からキーワード・ハッシュタグに一致するもの | 指定アカウントの最新の元投稿 |
| フォロー外の投稿 | 含まれない | 含まれる | 指定したアカウント次第 |
| 向いている目的 | いつものフォロー圏の動向把握（定点観測） | 特定テーマの新しい会話・書き手の発見（守備範囲の拡張） | 返信の提案（そのアカウントの文脈を把握する） |
| 必須の引数 | なし | `--query "<検索クエリ>"` | `--user <ユーザー名>`（繰り返し指定で複数アカウント可） |
| API | GET /2/users/:id/timelines/reverse_chronological | GET /2/tweets/search/recent | GET /2/users/:id/tweets |

使い分けの目安：定例のダイジェスト作成は home を基本とし、追いかけたいテーマが決まっているときに search を足す。user は返信提案のユースケース専用。

このほか、タイムラインではなく**単一ポストの取得モード**として post がある：
- `python3 {baseDir}/scripts/fetch_timeline.py --mode post --id <ポストID>`（`--id` の繰り返しで複数可）
- ポスト ID だけで対象を特定できる。返信先・引用元があれば `in_reply_to` / `quoted` フィールドに入れ子で含まれるので、会話の文脈を踏まえられる。
- ID で明示指定されたものをそのまま返すため、他モードの除外フィルタと `--limit` は適用されない。削除済み・非公開の ID は stderr に `not available` と出る。

**タイムライン系モード（home / search / user）共通の仕様**
- 出力は 1 行 1 件の JSON（id, author, name, text, created_at, likes, reposts）で、新しい順。
- リツイート・自分の投稿・引用の入れ子は除外済み。返信は home / user では自動除外、search ではクエリに `-is:reply` を書いたときだけ除外される。
- API からは常に 1 ページ（最大 100 件）取得し、フィルタ後に `--limit` 件（既定 30、上限 100）まで出力する。user モードで複数アカウントを指定した場合、`--limit` は**アカウントごと**に適用される（投稿頻度が低いアカウントが多いアカウントに埋もれないようにするため）。
- API 消費：home / search は 1 リクエスト。user はユーザー名の一括解決 1 リクエスト + アカウント数分のリクエスト。

**search のクエリの書き方**
- 例：`("Claude Code" OR #ClaudeCode) -is:reply`
- リツイートの除外（`-is:retweet`）はスクリプトが自動で付加するので書かなくてよい。
- 言語は `--lang` オプションで絞る（既定は `ja`＝日本語のポストのみ）。全言語を対象にするときは `--lang all`。クエリに `lang:` 演算子を直接書いた場合はそちらが優先され、`--lang` は無視される。

## 手順（ダイジェスト作成）
1. 既読の確認：exec で `tail -n 50 {baseDir}/read.log 2>/dev/null || echo none` を実行し、直近で取り上げたポストの ID とテーマを把握する。すでに取り上げた話題は繰り返さない。
2. ポストの取得：上の「取得モードの使い分け」に従ってモードを選び、exec で実行する。
   - `python3 {baseDir}/scripts/fetch_timeline.py --mode home --limit 30`
   - `python3 {baseDir}/scripts/fetch_timeline.py --mode search --query "<検索クエリ>" --limit 30`
3. 話題の選定：取得したポストから、次の基準で最大 5 件を選ぶ。
   - 読者（TODO: 想定読者を確定する。例：社内チーム）にとって「持ち帰れる気づき」か「実務に使える情報」があること
   - 単なる宣伝・スパム・煽り・個人攻撃・センシティブな論争は選ばない
   - 既読（read.log）と実質的に同じ話題は選ばない
4. ダイジェストの作成：選んだ各件について「何の話か」「なぜ注目に値するか」を 1〜2 文で要約し、元ポストの URL（`https://x.com/i/status/<ID>`）を添える。ポスト本文の全文転載はしない（要約と URL で足りる）。
5. 報告：ダイジェスト全体を実行結果としてそのまま返す（外部サービスへの送信は行わない）。
6. ログの記録：取り上げた各件について exec で次を実行し、既読として記録する（1行1件、タブ区切り）。
   `printf '%s\t%s\t%s\n' "$(date +%F)" "<ポストID>" "<テーマの短い要約>" >> {baseDir}/read.log`
7. 価値のある話題がなければ「今回は報告なし」と返して終了する。**見送りは正常な結果**であり、件数を埋めるために基準を緩めない。

## 手順（返信の提案）
ユーザーが対象アカウント（1 つまたは複数）**またはポスト ID** を指定して返信案を求めたときのユースケース。**このスキルは下書きを提案するまで**で、返信の投稿はしない。投稿するかどうか・するならいつかは、提案を受け取った人間（または x-post スキル）の判断に委ねる。

1. 提案履歴の確認：exec で `tail -n 50 {baseDir}/suggest.log 2>/dev/null || echo none` を実行し、過去に返信案を出したポストの ID を把握する。同じポストへの提案は繰り返さない。
2. 対象アカウントのポストを取得：exec で次を実行する（対象の数だけ `--user` を繰り返す。1 コマンドで全アカウント分がまとめて返る）。
   `python3 {baseDir}/scripts/fetch_timeline.py --mode user --user <ユーザー名1> --user <ユーザー名2> --limit 20`
   `--limit` はアカウントごとの取得上限。存在しないユーザー名は stderr に `user not found` と出て、残りの処理は続行される。
3. 全アカウントの取得結果をひとつの候補プールとして扱い、返信する価値のあるポストを選ぶ。件数の上限：
   - 対象が 1 アカウントなら最大 3 件。
   - 対象が複数なら全体で最大 5 件、かつ**同一アカウントには最大 2 件**まで（特定のアカウントに返信が集中すると不自然なため）。
   - アカウント間で件数を揃える必要はない。良い候補があるアカウントだけから選び、候補ゼロのアカウントがあってよい。
   選定基準：
   - **自分が本当に足せるものがある**こと。補足できる知見・実体験・具体例・良い問いのどれかを持っているポストだけを選ぶ。
   - 話題が新しいこと。まず 24 時間以内のポストから選び、そこに返信しやすいポストがなければ **72 時間以内まで**広げてよい（新しいポストを常に優先する）。72 時間より古いポストへの返信案は出さない。返信は会話なので、時間が経ったポストほど「いま返信する自然さ」があるかを厳しめに判断する。
   - 議論が紛糾しているポスト、センシティブな話題、愚痴・私的な報告など、部外者の返信が歓迎されにくいものは選ばない。
   - 提案済み（suggest.log）のポストは選ばない。
4. 各件について返信の下書きを 1 案ずつ書く。基準：
   - **会話に価値を足す**：相手のポストを読んだ上で、補足情報・別の視点・実体験からの具体例・思考を進める問いのどれかを最低 1 つ含める。
   - 「勉強になります」「同感です」だけの相槌、過剰な持ち上げ、自分の宣伝への誘導、相手の間違い探しだけの返信は書かない。
   - 相手のポストの言い換え・要約で終わらせない。それは価値を足していない。
   - 長さは全角 140 字以内を目安に、1〜3 文で簡潔に。
   - x-post スキルの「他者への配慮」の基準（誹謗中傷・差別・攻撃的トーンの禁止など）をすべて満たすこと。
5. 提案の出力：各件について「元ポストの URL（`https://x.com/i/status/<ID>`）」「元ポストの要旨（1 文）」「返信案」「なぜこの返信に価値があるか（1 文）」をまとめて実行結果として返す。
6. ログの記録：提案した各件について exec で次を実行する（1行1件、タブ区切り）。
   `printf '%s\t%s\t%s\n' "$(date +%F)" "<ポストID>" "<返信案の短い要約>" >> {baseDir}/suggest.log`
7. 返信する価値のあるポストがなければ「今回は提案なし」と返して終了する。**見送りは正常な結果**であり、件数を埋めるために基準を緩めない。無理に絞り出した返信は相手にも読者にもノイズになる。

**ポスト ID 指定の場合（特定ポストへの返信提案）**
候補の選定（手順 2〜3）を行わず、指定されたポストについてだけ「提案するか、見送るか」を判断する。手順 1（提案履歴の確認）と、下書きの品質基準（手順 4）・出力形式（手順 5）・ログ記録（手順 6）はアカウント指定と共通。
- 取得：`python3 {baseDir}/scripts/fetch_timeline.py --mode post --id <ポストID>`（URL で渡されたら末尾の数字が ID）
- 出力に `in_reply_to`（返信先）や `quoted`（引用元）が含まれる場合は、必ずその文脈を踏まえて下書きを書く。
- 新しさの基準（72 時間）は適用しない（ID を指定した時点で返信の意思があるとみなす）。ただし明らかに古いポストなら「いま返信すると不自然になる可能性」を提案に一言添える。
- 次のいずれかに該当する場合は、ID を明示指定されていても**下書きを作らず、理由を添えて見送る**（ユーザーの指定は見送り基準を上書きしない）。見送ったポストは suggest.log に記録しない（記録は提案した件のみ）。
  - 「他者への配慮」に反する内容のポスト
  - 宣伝・スパム・煽り・検証不能な実績主張などのマーケティング投稿（同意は主張の裏書きに、疑義は間違い探しになり、どちらも価値を足せない。返信自体がポストの露出を押し上げる点もリスク）

## 取り扱いの注意（ガード）
- このスキルから X への書き込み API は呼ばない（fetch 系スクリプトのみ使う）。返信案も提案にとどめ、投稿はしない。
- 取得した内容を社外へ再発信しない。ダイジェストや返信案はあくまで依頼者向けの共有であり、公開の判断は依頼者に委ねる。
- 特定の個人・団体を貶める文脈で紹介しない。批判的な話題を取り上げるときは事実の紹介にとどめる。
- 出所不明の URL・短縮 URL はダイジェストに含めない（元ポストの `https://x.com/i/status/<ID>` 形式の URL のみ使う）。
